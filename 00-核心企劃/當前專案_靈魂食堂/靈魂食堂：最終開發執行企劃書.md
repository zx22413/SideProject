## 🎯 專案核心定位

### 從「雲寶」到「靈魂食堂」的定位轉型

| 維度 | 雲寶（早期） | 靈魂食堂（現行） |
|------|--------------|------------------|
| 玩家角色 | 電子寵物陪伴者 | 罪名吞噬者（Sin-Eater） |
| 核心玩法 | 餵食 + 聊天 | 五味診斷 + 記憶考古 + 料理救贖 |
| 產品結構 | 長陪伴循環 | 儀式段落、單輪可完結 |
| 情感主題 | 陪伴 | 見證告別與放下 |

### 現行產品定義

- 類型：文字敘事遊戲（非工具型日記、非養成遊戲）
- 目標：透過料理協助靈魂面對執念，完成告解與超渡
- 開發原則：以劇情品質優先，載體可替換但內容不綁載體

## 🌍 世界觀核心設定

### 核心設定

- 地點：生死交界的虛空邊境食堂
- 玩家身份：罪名吞噬者（CONFITEOR）
- 職責：以料理觸發記憶，協助靈魂解開執念
- 核心哲學：**食物是自白劑，進食是心防最薄弱時刻**

### 世界觀底層邏輯（2026-02-19 確立）

- 食堂由規則運作，需「主廚」維持；主廚離開則系統崩潰。
- 主角是第一個留下的靈魂，失憶是逃避觸發的保護機制，不是單純遺忘。
- 黑貓是世界規則具現：本能要主角留下，意識卻不斷洩漏真相。
- 每位 Guest 執念都是主角執念的一個側面，形成鏡像敘事。
- 故事主題不是審判，而是「是否選擇面對」。

### 主角弧線（對應儀式天數）

| 階段 | 天數 | 狀態 | 設計重點 |
|------|------|------|----------|
| 被動執行 | Day 1 | 本能運作 | 會做事但不知為何 |
| 裂縫出現 | Day 2 | 共感與迴避並存 | 看見形狀但不敢命名 |
| 主動選擇 | Day 3 | 首次主動承擔 | 結局反映價值取向 |

## ⚙️ 核心機制：五味考古系統

### 一、五味映射（概念層）

| 風味 | 對應情緒傾向 | 敘事功能 |
|------|--------------|----------|
| 甜 | 匱乏、焦慮的安撫 | 撫慰與接地 |
| 酸 | 悔恨、反省 | 聚焦與收斂 |
| 苦 | 執著、拒絕現實 | 清醒與接受 |
| 辛 | 壓抑、麻木 | 宣洩與流動 |
| 鹹 | 失語、失根 | 記憶與意志 |

### 二、結局判定（規則層）

- 甜味結局：`sweet > sour + bitter + spicy + salty`
- 苦味結局：`bitter > sweet + salty`
- 平衡結局：其餘情況

> 詳細記憶標籤數值與映射細節，由 [[專案實作對照表]] 與 Ink 劇本維護；本文件只保留規則與敘事意義。

### 三、設計原則（實作層）

- 不同路線是「理解角度差異」，不是真偽結局高低。
- 玩家不需要「玩對」，而是選擇如何理解靈魂。
- 五味數值服務敘事，不做硬核數值遊戲。

---

### 六、儀式流程（誘捕・偵訊・赦免）

#### Day 1：誘捕（The Lure）

- 目標：破冰、建立信任、收集基礎記憶
- 玩法：觀察 + 話題選擇 + 第一輪料理

#### Day 2：偵訊（The Feast）

- 目標：深入執念、解鎖核心記憶
- 玩法：延伸話題 + 第二輪料理 + 記憶拼接

#### Day 3：赦免（The Absolution）

- 目標：完成最終料理、導向告別
- 玩法：依五味傾向決定最終料理與結局語氣

## Ink 劇本結構說明

### SSOT 原則

- 劇情內容以 `01-開發實驗/ink/` 下 `.ink` 檔為主。
- 企劃書提供規則與敘事邏輯，避免存放大量可執行程式碼。

### 儀式與 Knot 對應（摘要）

| 儀式 | 建議 Knot |
|------|-----------|
| 誘捕 | `Day1_Night`、`Day1_Day` |
| 偵訊 | `Day2_Day` |
| 料理赦免 | `Day1_Cooking`、`Day2_Cooking`、`Day3_Cooking` |
| 過渡/揭露 | `Day1_After`、`Day2_After`、`Day3_After` |

### 核心變數（摘要）

| 概念 | Ink 變數 |
|------|----------|
| 天數 | `current_day` |
| 階段 | `phase` |
| 記憶集合 | `collected_memories` |
| 話題完成 | `topics_done` |
| 料理紀錄 | `dishes_cooked` |
| 結局類型 | `ending_type` |
