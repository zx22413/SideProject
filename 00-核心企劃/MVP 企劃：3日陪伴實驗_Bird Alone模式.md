## 1. 實驗目標 (Objective)

### 核心目標
驗證 **「情感連結」** 是否能成為比「數據紀錄」更強的留存驅動力。  
重點在於測試**「非 AI 的預設對話」**是否能建立足夠深的情感連結。

### V1.4 新增目標
- **驗證「預設對話庫」的有效性**：測試精心設計的腳本是否比 AI 生成更能打動人
- **驗證「狀態機記憶系統」**：測試規則驅動的記憶機制是否能建立「被記得」的感覺
- **驗證「外部網頁視覺回饋」**：測試動態樣貌是否能增強情感連結
- **驗證「3 天高密度策略」**：測試壓縮到 3 天是否能維持足夠的情感深度

### 🎯 驗證指標 (Success Metrics)

#### 量化指標
1. **留存率**：3 天實驗期間，使用者是否每天都回來互動？（目標：≥ 90%）
2. **互動完成率**：每天完成「餵食 + 聊天」的比例（目標：≥ 80%）
3. **網頁查看率**：點擊「查看狀態」進入外部網頁的比例（目標：≥ 70%）
4. **記憶感知率**：Day 2-3 時，用戶是否感知到「雲寶記得我」？（目標：≥ 70%）

#### 質化指標
1. **情感連結**：用戶是否對 [[雲寶]] 產生「專屬樹洞」的認知？
2. **對話滿意度**：用戶是否認為預設對話「有溫度」而非「死板」？
3. **視覺回饋滿意度**：用戶是否認為外部網頁的動態樣貌有助於建立情感連結？

## 2. 產品規格 (MVP Specs)

### 技術架構
採用 **LINE OA (遊戲大廳) + 外部網頁 (視覺呈現) + 預設對話庫 (非 AI)** 的三平台架構。

### ⚙️ 核心交互循環 (The Interaction Loop)

參考 [[企劃書與Bird Alone結合探討#3.1 核心循環]] 的設計：

| 階段 | 用戶動作 (User) | 系統反應 (System) | 設計意圖 (UX Goal) |
|:-----|:---------------|:----------------|:------------------|
| **觸發** | (收到推播)<br>「主人主人～餓餓...」 | **LINE OA 推播**：<br>顯示「今日任務」+ 選擇題 | 喚醒用戶，建立「被需要」的感覺 |
| **行動** | 點擊「餵食雲寶」或「與雲寶聊天」 | **預設對話回應**：<br>根據狀態機匹配對應對話 | 提供即時、穩定的情感回饋 |
| **酬賞** | 查看外部網頁 | **動態樣貌更新**：<br>雲寶根據情緒變色/變形 | 視覺化情感連結，增加成就感 |
| **投入** | Day 2-3 時 | **記憶系統觸發**：<br>雲寶提到 Day 1 的選擇 | 建立「被記得」的感覺，強化情感連結 |

## 3. 核心機制 (Core Mechanics)

### 機制 A：預設對話庫系統 ⭐ **V1.4 核心機制**

- 參考 [[企劃書與Bird Alone結合探討#5.2 避免重複的對話設計模式]] 的「桶」系統
- 不使用 AI，所有對話都是預先寫好的腳本
- 透過「桶系統」+ 狀態機匹配，創造「看似動態」的對話體驗

### 機制 B：狀態機記憶系統 ⭐ **V1.4 核心機制**

**設計理念**：
- 參考 [[企劃書與Bird Alone結合探討#4.2 狀態機設計]] 的有限狀態機
- 不使用 AI 記憶，而是用資料庫記錄用戶的選擇和情緒
- 透過變量插值，創造「被記得」的感覺

**核心變量架構**：

```javascript
// 用戶狀態（儲存在 localStorage 或 Firebase）
const userState = {
  // 基本資訊
  userId: "xxx",
  startDate: "2025-01-XX",
  
  // 進度變量
  day: 1, // 1-3
  lastInteraction: "2025-01-XX 21:00",
  timeOfDay: "evening", // morning / afternoon / evening / night
  
  // 情緒變量
  currentMood: "happy", // happy / sad / angry
  moodHistory: [
    { day: 1, mood: "happy", time: "2025-01-XX 21:00" },
  ],
  
  // 關係變量（簡化版）
  trustLevel: 30, // 0-100（根據互動次數和深度計算）
  intimacy: 20, // 0-100（根據分享的深度計算）
  
  // 歷史標籤（記憶系統）
  tags: ["拌拌派"], // 從選擇題中提取
  memoryPoints: [
    { day: 1, choice: "A", tag: "拌拌派", question: "咖哩怎麼吃？" },
  ],
  
  // 互動記錄
  interactions: [
    { day: 1, action: "feed", mood: "happy", response: "day1_feed_happy_1" },
  ],
};
```

**記憶系統運作方式**：

1. **標籤提取**：當用戶完成選擇題時，系統記錄對應的標籤
   - 例如：Day 1 選擇「拌拌派」→ 記錄 `tags: ["拌拌派"]`

2. **記憶觸發**：Day 2-3 時，系統根據標籤匹配對應的記憶對話
   - 例如：如果 `tags` 包含「拌拌派」→ 匹配 `day2_memory_bunbun`

3. **變量插值**：在對話中使用變量，創造個人化感覺
   - 例如：「主人昨天說咖哩都拌拌吃～雲寶記住了！」

### 機制 C：外部網頁動態樣貌 ⭐ **V1.4 核心機制**

**設計理念**：
- 參考 Bird Alone 的視覺呈現方式
- 視覺為主，文字為輔
- 根據用戶的選擇和情緒，動態更新雲寶的樣貌

**網頁功能設計**：

1. **顯示雲寶的動態樣貌**
   - 根據 `day`、`mood`、`tags` 動態更新
   - 加入簡單動畫（飄動、發光、下雨等）

2. **顯示雲寶的狀態**
   - 情緒狀態（開心 ☀️ / 難過 💧 / 生氣 🌶️）
   - 進度（Day X/3）
   - 記憶標籤（例如：「主人是拌拌派」）

3. **顯示「小語語錄」**
   - 類似 Bird Alone 的訊息
   - 例如：「要記住，你隨時都可以過來。雲寶在等你。」

**技術實現**：


**動態效果**：
- 根據 `mood` 變色（開心→金黃色，難過→深藍灰色，生氣→暗紅色）
- 根據 `day` 變形（Day 1→小圓球，Day 2→長手手，Day 3→最終形態）
- 根據 `tags` 顯示特殊效果（例如：「拌拌派」→ 雲寶會「攪拌」動畫）

### 機制 D：簡化行動系統 ⭐ **V1.4 核心機制**

**設計理念**：
- 只保留 3 個核心行動：餵食、聊天、查看狀態
- 降低認知負擔，聚焦核心價值

**行動設計**：

| 行動名稱 | 消耗點數 | 設計意圖 | LINE OA 實作方式 |
|:--------|:--------|:--------|:----------------|
| **餵食雲寶** | 1 點 | 核心機制，建立「情緒即食糧」 | 圖文選單按鈕 → 選擇情緒 → 預設對話回應 |
| **與雲寶聊天** | 1 點 | 保留「情緒樹洞」核心價值 | 圖文選單按鈕 → 選擇題 → 預設對話回應 |
| **查看狀態** | 0 點 | 提供「查看」滿足感，不消耗點數 | 圖文選單按鈕 → 外部網頁連結 |

**行動點數分配**：
- 每天 2 點（建議）
  - 1 點：餵食雲寶（必做）
  - 1 點：與雲寶聊天（可選）
  - 查看狀態：0 點（無限查看）

## 4. 3 天高密度劇情腳本

### Day 1：誕生 → 第一次餵食 → 建立連結

**推播文案**：
「主人主人～我是雲寶！剛出生，什麼都不知道... 餓餓...  
今天發生了什麼事？快點來餵我吃東西！☁️✨」

**劇情節點**：
1. **開場**：雲寶剛出生，對世界一無所知
2. **第一次餵食**：用戶選擇情緒（開心/難過/生氣），雲寶第一次「吃」到情緒
3. **第一次聊天**：選擇題「咖哩怎麼吃？」（A. 拌拌派 / B. 分開派）
4. **查看狀態**：網頁顯示雲寶從「小圓球」變成「有表情的雲朵」（根據餵食的情緒變色）

**對話庫重點**：
- 建立「被需要」的感覺
- 第一次互動要有「驚喜感」
- 強調「情緒即食糧」的概念

### Day 2：成長 → 發現變化 → 建立記憶

**推播文案**：
「主人！雲寶昨天吃飽飽，今天身體變得不一樣了！你看你看！  
今天也要繼續餵我喔！☁️✨」

**劇情節點**：
1. **開場**：雲寶記得昨天的事（記憶系統觸發）
2. **第二次餵食**：用戶選擇情緒，雲寶提到 Day 1 的記憶
3. **第二次聊天**：選擇題「累爆時會先做什麼？」（A. 放空派 / B. 儀式派）
4. **查看狀態**：網頁顯示雲寶長出手手 / 變色 / 發光（根據 Day 1-2 的選擇）

**對話庫重點**：
- 強調「記憶」機制（雲寶記得昨天的事）
- 視覺變化要明顯（手手、變色、發光）
- 建立「成長感」

### Day 3：進化 → 告別 → 情感昇華

**推播文案**：
「主人！雲寶感覺身體熱熱的... 好像要變身了！快來看！  
這是最後一天了... 雲寶想跟主人多說說話... ☁️✨」

**劇情節點**：
1. **開場**：雲寶感覺要變身了（預告結局）
2. **第三次餵食**：用戶選擇情緒，雲寶提到 Day 1-2 的記憶
3. **第三次聊天**：選擇題「這三天跟雲寶相處比較像什麼？」（A. 樹洞模式 / B. 陪伴模式）
4. **查看狀態**：網頁顯示雲寶最終形態（根據 3 天選擇決定：星星 / 雨水 / 彩虹）
5. **結局**：根據 3 天選擇決定結局對話

**對話庫重點**：
- 情感高潮（告別、感謝）
- 結局要有「儀式感」
- 網頁最終形態要「美」且「值得截圖分享」

## 5. 技術架構 (Technical Architecture)

### 前端架構

**LINE OA（遊戲大廳）**：
- 圖文選單（3 個按鈕：餵食、聊天、查看狀態）
- 關鍵字自動回應（根據通關密語匹配預設對話）
- 推播通知（每天定時推播）

**外部網頁（視覺呈現）**：
- HTML + CSS + JavaScript
- 根據 URL 參數（`?day=1&mood=happy&tags=拌拌派`）動態更新
- 資料儲存：localStorage（簡單版）或 Firebase（跨裝置同步）

### 後端架構（可選）

**如果不需要跨裝置同步**：
- 完全前端，用 localStorage

**如果需要跨裝置同步**：
- Firebase / Supabase（簡單的資料庫）
- 記錄用戶的選擇和情緒

### 對話庫管理

**建議工具**：
- JSON 檔案（簡單版）
- Google Sheets + API（方便非技術人員編輯）
- 專用的對話庫管理工具（未來擴展）

## 6. 內容生產策略

### 對話庫規模

**總字數**：約 3,000-5,000 字

**分配建議**：
- Day 1：1,000-1,500 字
  - 餵食回應（開心/難過/生氣各 10-15 種變體）
  - 選擇題回應（A/B 各 5-10 種變體）
  - 記憶回應（Day 2 提到 Day 1 的變體）
- Day 2：1,500-2,000 字
  - 餵食回應（同上 + 提到 Day 1 記憶）
  - 選擇題回應（同上）
  - 記憶回應（Day 3 提到 Day 1-2 的變體）
- Day 3：1,500-2,000 字
  - 餵食回應（同上 + 提到 Day 1-2 記憶）
  - 選擇題回應（同上）
  - 結局對話（根據選擇決定）

### 寫作原則

參考 [[企劃書與Bird Alone結合探討#5.1 建立「無辜賢者」的角色聲音]]：

1. **保持「史伊式」語氣**：
   - 簡單、直率的詞彙
   - 幼兒詞彙、疊字、直接行動
   - 不要說教，不要給建議

2. **建立「桶」系統**：
   - 每個情境準備 10-15 種變體
   - 避免重複，延長內容消耗壽命

3. **處理輸入**：
   - 關鍵決策用選擇題（A/B）
   - 情感宣洩用自由文本（但系統不解析，只根據字數和速度給予模糊反饋）

## 8. 風險與應對 (Risks & Mitigation)

### 風險 A：預設對話顯得「死板」

**問題描述**：
- 用戶可能覺得預設對話缺乏「靈活性」
- 與 AI 對話相比，可能顯得「不夠智能」

**應對策略**：
1. **建立「桶」系統**：每個情境準備 10-15 種變體，增加變化
2. **加入隨機性**：根據時間、標籤、關係變量組合，創造「看似動態」的體驗
3. **強調「穩定性」**：預設對話的優勢是「穩定」和「深度」，而非「靈活」

### 風險 B：記憶系統不夠「智能」

**問題描述**：
- 規則驅動的記憶系統可能不如 AI 記憶「自然」
- 用戶可能察覺到「這是預設的」

**應對策略**：
1. **變量插值**：在對話中使用變量，創造個人化感覺
2. **模糊化回應**：記憶回應要「通用」而非「具體」，讓用戶自己腦補
3. **強調「被記得」的感覺**：重點不是「記得多準」，而是「被記得」的感覺

### 風險 C：外部網頁開發成本

**問題描述**：
- 外部網頁需要額外的開發成本
- 動態效果可能增加複雜度

**應對策略**：
1. **簡化設計**：先做最簡單的版本（靜態圖片 + 簡單動畫）
2. **漸進式開發**：先完成核心功能，再優化視覺效果
3. **使用現成工具**：考慮使用現成的網頁模板或框架

## 💡 備忘錄 (Notes)

### 關於 AI vs 非 AI

- **AI 的優勢**：靈活性、個人化、開放式對話
- **非 AI 的優勢**：穩定性、深度、可控性、成本低
- **結論**：對於 3 天高密度策略，非 AI 的預設對話庫更適合

### 關於 Bird Alone 模式

- **核心特徵**：預設對話、選擇題、視覺為主、規則驅動
- **適用場景**：短期實驗、高密度劇情、情感深度
- **不適用場景**：長期陪伴、開放式對話、複雜互動

### 關於外部網頁

- **設計重點**：視覺為主，文字為輔
- **技術重點**：簡單、穩定、快速載入
- **未來擴展**：可以加入更多互動（撫摸、音樂等）
