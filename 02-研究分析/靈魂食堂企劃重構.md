# **【專案代號：CONFITEOR】靈魂食堂最終開發執行企劃書：神聖敘事與五味考古機制的深度融合**

## **執行摘要：數位時代的儀式重構**

本文件旨在為《靈魂食堂》（Soul Cafeteria，專案代號：CONFITEOR）的開發提供一份詳盡、具備高度可執行性且理論基礎深厚的最終企劃書。本企劃案係針對先前三份獨立提案——側重氛圍的「告解世界觀」、側重技術的「AI 驅動標籤系統」、側重邏輯的「五味診斷機制」——進行深度辯證與有機融合的成果。

鑑於《架構診斷報告》中指出的核心矛盾——即「V4.0 的宏大願景與 V1.4 的落後代碼」之間的斷裂，以及 LINE OA 平台在「電子寵物」與「日記工具」定位上的產品裂痕，本企劃書確立了全新的開發範式：**「數位禮儀（Digital Liturgy）」**。我們將捨棄傳統模擬經營遊戲中對物理操作（如切菜、烹飪動作）的執著，轉而擁抱「文本考古」與「概念煉金術」。

本專案的核心願景在於定義並實踐\*\*「溫暖但具有重量感（Warm Weight）」\*\*的獨特美學。在這裡，「溫暖」並非廉價的療癒或逃避，而是如同暴風雨夜中修道院爐火般的庇護感；「重量」則來自於玩家作為「罪名吞噬者（Sin-Eater）」所承擔的道德責任——透過料理去消化那些生者無法承受、死者無法帶走的遺憾。這是一場關於記憶、味覺與救贖的心理實驗，旨在利用 LINE 這種高頻低壓的通訊軟體，構建出一個深邃的閾限空間（Liminal Space）。

**第一部分：核心願景——定義「溫暖但具有重量感」的體驗**

## **1.1 從「審判」到「見證」的範式轉移**

在傳統的生死題材遊戲（如《Papers, Please》或《Death and Taxes》）中，遊戲張力往往源自於「審判（Judgment）」——玩家掌握生殺大權，根據規則決定靈魂的去留。這種機制雖然能產生高強度的戲劇衝突，但本質上是冷酷且功利主義的。它要求玩家將靈魂視為待處理的「案件」，而非具備情感厚度的「主體」。

《靈魂食堂》的核心願景要求我們徹底屏棄這種上帝視角的傲慢。我們提出的新範式是\*\*「見證（Witnessing）」\*\*。

### **1.1.1 權力的讓渡與共情的建立**

在新的設計中，玩家扮演的「罪名吞噬者」並非高高在上的法官，而是服務者。這種權力結構的倒置是建立「溫暖」的基礎。

* **審判是權力的行使**：它預設了對錯的標準，並將玩家置於道德高地。  
* **見證是權力的讓渡**：它要求玩家放下評判，單純地「在場」。如同安寧病房的陪伴者，我們不試圖修正死亡，而是陪伴對方走完最後一程。

這種轉變將遊戲的張力來源從「害怕做錯決定的恐懼」轉化為「渴望理解對方的責任」。當玩家意識到自己手中的料理是該靈魂解脫桎梏的唯一鑰匙，且這把鑰匙必須通過深刻的理解（而非猜謎）來鍛造時，遊戲便產生了「重量感」。

### **1.1.2 不可逆的同理心**

「重量」的另一個來源是\*\*「不可逆性」\*\*。在我們的設計中，靈魂的離去是必然的。我們不是在拯救他們復活，而是在送別。每一次成功的「料理治癒」，都意味著玩家將永遠失去這位朋友。這種「為了對方的幸福而必須放手」的矛盾情感，構成了極具文學深度的「苦樂參半（Bittersweet）」體驗 1。這與《Spiritfarer》的設計哲學不謀而合，但在 LINE 的碎片化敘事中，這種離別顯得更為日常且真實——就像一位久未聯繫的朋友突然從聊天列表中消失。

## **1.2 溫暖的物理性定義：Hygge 與庇護所**

我們所定義的「溫暖」，必須具備物理性的感知基礎。參考丹麥文化中的 **Hygge（舒適愜意）** 概念，溫暖不僅僅是溫度，更是一種「對抗外部虛無的安全感」。

### **1.2.1 暴風雨中的爐火**

為了凸顯溫暖，我們必須先創造寒冷。遊戲的背景設定在一個永恆下雨的虛空邊境。玩家的食堂是這片無盡灰暗中唯一的光源。

* **聽覺對比**：背景音效（Soundscape）將由窗外持續的雷雨聲與室內柴火爆裂聲、湯鍋燉煮的咕嘟聲交織而成。這種內外的聽覺對比，會在玩家潛意識中建立起「這裡很安全」的心理暗示 2。  
* **視覺對比**：介面設計將大量使用深色調（哥德式的黑、深褐），而食物插畫則採用吉卜力風格（Ghibli-style）的高飽和度、高光澤畫法。食物在畫面中發光，不僅是視覺焦點，更是情感的熱源。

## **1.3 敘事安全閥：準備與緩衝**

當遊戲涉及死亡、創傷與遺憾時，過度的沉重可能會讓玩家產生防禦機制（想要關閉遊戲）。為了維持「溫暖」的基調，我們引入了\*\*「敘事安全閥（Narrative Safety Valve）」\*\*機制。

### **1.3.1 廚房視角的抽離感**

在靈魂進行深度告解（可能涉及暴力、背叛等沈重話題）之後，系統會強制切換回「廚房視角（Kitchen View）」。這不僅是玩法的切換，更是情緒的緩衝區。在廚房裡，玩家專注於食材的挑選與合成，這種邏輯性的、規律性的操作（Keyword Alchemy），能讓玩家從強烈的情感共鳴中暫時抽離，整理情緒，避免情感過載（Emotional Burnout）。這就像心理諮商師在諮商結束後需要寫病歷報告一樣，是一種必要的專業抽離。

**第二部分：邊境食堂世界觀——視覺意象與儀式架構**

## **2.1 空間美學：虛空中的西多會修道院**

遊戲的舞台——「邊境食堂」，絕非普通的日式居酒屋。它是一個位於生死閾限（Liminality）的神聖空間。我們採用\*\*哥德復興（Gothic Revival）**與**西多會（Cistercian）**建築語彙，結合**大正浪漫（Taisho Roman）\*\*的細節裝飾，創造出一種既莊嚴又帶有懷舊溫度的獨特美學 3。

### **2.1.1 建築佈局：告解室結構**

食堂的空間佈局直接挪用了天主教告解室（Confessional Booth）的物理結構 5。

* **格柵（The Lattice）**：廚房與用餐區之間並非開放式櫃台，而是一道厚重的深色木製格柵。玩家（主廚）在明亮的火爐前操作，而客人（靈魂）則隱沒在格柵另一側的陰影中。  
* **視覺隱喻**：這種設計創造了權力的不對等與保護。光線照亮了食物，卻隱藏了食客的面容。這讓靈魂感到安全，因為他們不需要在強光下暴露自己的羞恥或脆弱。對於玩家而言，這種「隔著簾幕的窺視感」增加了神秘感與探索慾。

### **2.1.2 永恆之火與環境氛圍**

* **煉獄之火（Purgatory Fire）**：廚房中央的爐火並非普通的紅橘色，而是呈現幽藍色。這暗示了此處的火具有靈性屬性，它不燒毀物質，而是提煉情感 2。  
* **窗外的虛空**：透過高聳的彩色玻璃窗（上面繪製的不是聖人，而是食材的幾何圖騰），玩家看不到風景，只能看見流動的迷霧與永恆的雨幕。這場雨與現實世界的天氣同步（若技術允許）或隨劇情波動，增強了孤獨共感。

## **2.2 角色定位：罪名吞噬者（The Sin-Eater）**

玩家的角色代號為「CONFITEOR」（拉丁語：我告解），職位是\*\*「罪名吞噬者」\*\*。這是一個源自英國與威爾斯民俗傳說的概念，我們將其進行了現代化與神學化的重構 6。

### **2.2.1 民俗原型與現代轉化**

在歷史上，罪名吞噬者是社會邊緣人（Pariah），他們通過在屍體前吃下麵包與喝下愛爾啤酒，來吸收死者未竟的罪孽，使其靈魂得以安息。

* **轉化**：在《靈魂食堂》中，我們去除了「被詛咒」的負面標籤，保留了「承擔」的核心精神。玩家不再是為了幾枚銅板而食罪的乞丐，而是主動選擇承擔他人痛苦的\*\*「靈魂藥劑師」\*\*。  
* **形象設計**：玩家角色的視覺形象（僅在反射或過場中出現）身著改良式的雙排扣廚師服，外罩一件神父的黑袍（Cassock），領口掛著一條紫色的聖帶（Stole），象徵懺悔與補贖。腰間掛著的不是刀具，而是一串由乾燥八角、肉桂與辣椒串成的念珠 2。

### **2.2.2 雙重職責**

1. **主廚（Chef）**：負責將記憶的碎片（Tags）合成為可食用的料理。這需要邏輯與感性的平衡。  
2. **聽告解者（Confessor）**：負責在靈魂進食時，捕捉其潛意識洩漏的語言，並給予回應。

## **2.3 儀式流程：從誘捕到赦免**

遊戲的互動流程被設計為一套嚴謹的\*\*「數位禮儀（Digital Liturgy）」\*\*。每一個步驟都帶有儀式感，而非單純的遊戲操作。

### **2.3.1 第一階段：誘捕（The Lure）**

靈魂在剛進入食堂時，往往處於迷茫、防禦或失憶狀態。他們唯一的共同點是「飢餓」。這種飢餓並非生理上的，而是本體論上的匱乏。玩家必須利用「氣味」作為誘餌。

* **機制**：玩家選擇基礎香料（如「焦糖」、「洋蔥」），系統描述其香氣如何飄過格柵。這利用了心理學上的\*\*「口腔期回歸（Oral Stage Regression）」\*\*原理——進食讓靈魂回歸到嬰兒般的依賴狀態，從而卸下心理防線 2。

### **2.3.2 第二階段：偵訊（The Feast）**

當靈魂開始進食，遊戲進入「偵訊」模式。這不是傳統的問答，而是觀察。

* **蒸汽關鍵字（Steam Keywords）**：靈魂在咀嚼時無法說話，但他們的潛意識會隨著食物的熱氣升騰，幻化為模糊的文字（Keywords）漂浮在空中。例如，吃下辛辣料理時，可能會浮現「火」、「尖叫」、「紅色」等詞。玩家必須點擊採集這些詞彙，作為後續對話的籌碼。

### **2.3.3 第三階段：赦免（The Absolution）**

餐後的洗碗環節是儀式的終結。

* **洗碗即修行**：參考修士羅倫斯（Brother Lawrence）「上帝行走於鍋碗瓢盆之間」的神學觀，洗碗被視為神聖的淨化儀式 9。  
* **操作**：玩家透過滑動螢幕擦拭盤子上的油污。每一塊污漬的消失，都伴隨著清脆的音效與靈魂的一句心聲回放（Voiceover）。如果盤子洗淨後光亮如新，代表靈魂已無保留；若殘留頑垢，則代表仍有謊言。

### **3.1.1 診斷邏輯：表層需求 vs. 深層需求**

玩家的挑戰在於識別靈魂的「不可靠敘述」。靈魂往往會根據生前的習慣點餐（表層需求），但這往往掩蓋了他們靈魂真正缺乏的元素（深層需求）。

* **案例**：一位過勞死的上班族點了「最貴的和牛牛排」（表層：追求世俗成功，屬**甜/土**的過剩）。  
* **觀察**：玩家發現他不斷揉著胃部，且眼神渙散（診斷：土行失調導致的焦慮，且缺乏水行的意志）。  
* **處方**：玩家若照單全收，靈魂吃完只會感到更空虛。正確的處方應是「梅子茶泡飯」——\*\*酸（木）\*\*剋制過剩的土（焦慮），\*\*鹹（水）\*\*補足根源記憶，**熱湯**提供撫慰。

## **3.2 標籤考古系統：關鍵字煉金術**

鑑於技術限制（無法實作物理烹飪），我們採用\*\*「關鍵字標籤（Tag）採集」\*\*作為核心玩法。這將「閱讀」轉化為一種「採集（Foraging）」行為 12。

### **3.2.1 採集機制（Foraging）**

在對話介面（LINE Flex Message）中，靈魂的台詞會包含特定的高亮詞彙。

* **文本**：「那年冬天的**雪**下得很大，但我心裡卻有一團**火**。」  
* **操作**：玩家點擊「雪」，系統蒐集到 Tag: Cold；點擊「火」，蒐集到 Tag: Anger/Heat。  
* **雜訊過濾**：並非所有高亮詞都是有用的。有些是靈魂的自我防衛或謊言（如「我一點都不在乎」）。採集這些「雜質標籤（Impurity Tags）」會導致料理出現異味，觸發「不和諧分支（Dissonant Branch）」。

### **3.2.2 合成機制（Alchemy）**

進入廚房介面後，玩家面對的不是鍋碗瓢盆，而是一個類似《密教模擬器（Cultist Simulator）》的抽象合成台。

* **插槽（Slots）**：  
  1. **基底（Base）**：決定料理的型態（如米飯、麵條、湯）。對應靈魂的「現狀」。  
  2. **調味（Flavor）**：決定五味屬性（酸甜苦辣鹹）。對應靈魂需要的「藥方」。  
  3. **火候（Heat）**：決定處理方式（慢燉、快炒、冰鎮）。對應時間的「流速」。  
* **合成範例**：  
  * Tag: Childhood (Sweet/Base) \+ Tag: Regret (Sour/Flavor) \+ Tag: Long Wait (Simmer/Heat)  
  * **結果**：**「回憶苦甜燉肉」**（Bittersweet Stew）。這道菜能同時撫慰焦慮並引導反省。

**第四部分：三日敘事範例——失憶老裁縫的救贖**

為了展示上述機制的實際運作，以下以「失憶的老裁縫（Mr. Needle）」為例，演示一個完整的**3日敘事循環（3-Day Loop）**。

角色設定：Mr. Needle，生前是一流的西裝裁縫，死於阿茲海默症引發的低溫症。  
核心創傷：忘記了自己死前正在為女兒縫製婚紗，誤以為自己一事無成。  
五味診斷：需要鹹（水）來喚醒深層記憶，需要溫熱來對抗死亡時的寒冷。

## **Day 1：味道誘發（The Lure）**

* **情境**：Mr. Needle 裹著破舊的大衣，在食堂門口瑟瑟發抖。  
* **對話**：「冷……好冷。這裡也是雪山嗎？我的手……我的手動不了了。給我一點……熱的東西。也許……酒？」  
* **玩家操作**：  
  * **錯誤選擇**：給予烈酒（辛/火）。這會讓他身體發熱，但加劇混亂（火剋金，金生水，導致思緒更混亂）。

為了實現上述體驗，我們必須在 LINE OA \+ Google Sheets (GAS) 的技術框架下，構建一套輕量但強大的狀態機。

## **5.1 核心數據結構：userState 六大欄位**

我們將 Google Sheet 作為後端資料庫，userState 表單是核心。針對架構診斷報告的要求，我們定義以下 6 個關鍵欄位及其變動邏輯：

| 欄位名稱 (Column) | 資料類型 | 功能描述 | 變動邏輯與觸發點 |
| :---- | :---- | :---- | :---- |
| **A: user\_id** | String | LINE User ID (Primary Key) | 用戶首次加入時生成。 |
| **B: currentGuest** | String | 當前接待的靈魂 ID (e.g., "G01\_Needle") | 完成上一位靈魂超渡後，系統自動指派下一位。 |
| **C: progressDay** | Integer | 當前靈魂的劇情天數 (1-3) | 每日 00:00 伺服器時間檢查後解鎖，或完成當日關鍵任務後推進。 |
| **D: currentMood** | JSON | 當前靈魂的情緒狀態 (五行數值) | { "Wood": 20, "Fire": 50, "Earth": 10, "Metal": 80, "Water": 30 }。 每次料理評價後，根據料理屬性進行加減運算。 |

## **5.2 狀態機邏輯 (State Machine Logic)**

由於 LINE 是無狀態（Stateless）的通訊協定，我們必須在 GAS 中實作狀態機來管理對話流。

### **5.2.1 狀態定義**

* STATE\_LOBBY: 玩家在食堂大廳，可查看圖鑑或等待新客人。  
* STATE\_TALK: 進入對話模式，系統推送帶有 Postback Action 的 Flex Message。  
* STATE\_COOK: 進入廚房模式，玩家選擇 Tags 進行合成。  
* STATE\_EVAL: 系統判定料理結果，更新 currentMood 並回傳反應。

### **5.2.2 關鍵代碼邏輯 (Pseudo-code)**

JavaScript

function handlePostback(event) {  
  const userId \= event.source.userId;  
  const data \= JSON.parse(event.postback.data);  
    
  // 1\. Tag 採集邏輯  
  if (data.action \=== "collect\_tag") {  
    let userState \= getUserState(userId);  
    userState.collectedTags.push(data.tagId);  
    updateSheet(userId, userState);  
    return replyFlexMessage(event.replyToken, "TagCollectedTemplate", { tag: data.tagName });  
  }

  // 2\. 料理合成邏輯  
  if (data.action \=== "cook") {  
    const selectedTags \= data.tags; // e.g., \["Cold", "Honey"\]  
    const dishResult \= calculateDish(selectedTags); // 查表法  
      
    // 更新情緒數值  
    let mood \= getCurrentMood(userId);  
    mood \= applyDishEffect(mood, dishResult.attributes);  
      
    // 判定是否觸發回憶殺  
    if (checkProustEffect(mood, dishResult)) {  
       return replyNarrative(event.replyToken, dishResult.memoryCutscene);  
    } else {  
       return replyNarrative(event.replyToken, dishResult.normalReaction);  
    }  
  }  
}

## **5.3 限制與解決方案**

* **LINE Flex Message 互動限制**：LINE 不支援拖曳操作。  
  * **解決方案**：採用「點擊選擇（Tap to Select）」機制。廚房介面展示已收集的 Tags 列表（Carousel 形式），玩家點擊 Tag 使其進入「鍋中（Selected State）」，再次點擊取消。  
* **圖片生成成本**：  
  * **解決方案**：不進行實時 AI 生成。所有靈魂的塔羅牌狀態（正位/逆位/釋懷）與核心料理插畫，皆預先使用 Midjourney 生成並存儲於圖床，遊戲中僅進行調用。
